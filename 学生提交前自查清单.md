# 📋 R2G 实验 · 提交前自查清单

> 提交前对着这份清单逐项确认。机器能查的交给脚本，这份清单帮你确认"人才能确认的事"。
> 全部打勾再打包提交，能省掉绝大多数被退回/标红的麻烦。

---

## 第一步：跑机器自检（必做）

```bash
python3 r2g-rtl2gds/scripts/teaching/check_my_case.py --teaching-root /home/zhz/r2g_teaching
```

- [ ] 末尾显示 **"自检通过：没有会被标红的项。可以提交。"**
- [ ] 没有出现**红色**项（红色 = 严重问题，必须修，见下方对照表）
- [ ] 有**黄色**提示也没关系，但看一眼是不是"建议修正"类（如路径写法），顺手改掉更好
- [ ] **灰色**提示（`output_superseded` / `repo_root_unresolved`）完全无害，忽略即可

## 第二步：5 个 design 是否都齐（必做）

- [ ] 一共交了 **5 个 design**，每个在 `cases/<design>/` 下
- [ ] design 目录名是**顶层模块名**（如 `csr`），**不是 RTL 目录名**（如 `PYGMY_V32I_rtl_csr`）
- [ ] 没有空壳目录（如果有用错名字建出来的空目录，删掉）

每个 design 的 `cases/<design>/` 下应有：

- [ ] `CASE_STATE.md`
- [ ] 各 stage 的报告（`stage1_*`、`stage2_orfs`、`stage3_drc_lvs_rcx`、`stage4_labels`）
- [ ] **大产物**：`stage2_orfs/6_final.gds`、`6_final.def`、`6_final.odb`；
      `stage3_drc_lvs_rcx/6_final.spef`（若 RCX 跑出）
- [ ] **4 个 label CSV**：`stage4_labels/` 下 `wirelength` / `cell_congestion` /
      `timing_features` / `ir_drop`
- [ ] **8 个 feature CSV**：`stage4_features/` 下 `metadata` / `nodes_gate` / `nodes_net` /
      `nodes_iopin` / `nodes_pin` / `edges_iopin_net` / `edges_pin_net` / `edges_gate_pin`

## 第三步：CASE_STATE 关键字段（必做）

打开每个 `CASE_STATE.md` 确认：

- [ ] `platform: nangate45`（不能是别的平台，否则违反红线 3）
- [ ] `gds_path` / `def_path` / `odb_path` / `spef_path` 四个字段用
      **`<teaching_root>/cases/<design>/...`** 开头
      （**不要**写成 `<repo>/...`，否则大产物不受防篡改保护，会被标
      `case_state_path_should_be_teaching_root`）
- [ ] 没有写死机器绝对路径（不出现 `/home/...`、`/data1/...` 等）
- [ ] 失败/阻塞如实记录（如 `lvs_status: LVS_BLOCKED_MISSING_RULES`、
      `LVS_NOT_RUN`）——**如实记录不扣分**，不要为了好看改成无证据的 PASS

## 第四步：没有手工动过产物/账本（必做）

- [ ] 所有产物都是 `run_stage.sh` 跑出来的，**没有手工复制/移动/下载**任何
      `.gds/.def/.odb/.spef/.csv` 到 `cases/` 里
      （手搬的产物没有账本背书，会被标 `artifact_not_in_ledger`）
- [ ] 没有手动编辑过 `run_ledger.jsonl`、各 stage 报告、或任何已生成的产物
- [ ] 如果中途 `git pull` 更新过脚本，已经**重跑**了受影响的 stage
- [ ] 如果某个产物缺失，是**重跑对应 stage** 补的，不是手动拷进去的

## 第五步：打包提交（必做）

**进入 `r2g_teaching/` 内部再打包**，让压缩包里不含顶层 `r2g_teaching/` 目录
（这样老师解压后能直接批改，不用手工调层级）：

```bash
cd /home/zhz/r2g_teaching
zip -r ../姓名_学号.zip .            # 注意：先 cd 进目录，再打包当前目录「.」
```

- [ ] 是**进入 `r2g_teaching/` 后打包 `.`**，不是在外面 `zip -r 学号.zip r2g_teaching/`
      （后者会多一层目录，老师批改时层级会差一级）
- [ ] 解压自检一下：`unzip -l 姓名_学号.zip | head` —— 顶层应直接看到 `cases/`、
      `TEACHING_POLICY.md`、`run_ledger.jsonl`，**而不是** `r2g_teaching/cases/...`
- [ ] 压缩包命名为 **`姓名_学号.zip`**（如 `张三_2024XXXXX.zip`）
- [ ] 包含完整 **5 个 design** 的 `cases/`、`TEACHING_POLICY.md`、`run_ledger.jsonl`
- [ ] 没有把 `<repo>/.../backend/RUN_*/` 那些 50–150M 的 ODB 中间产物打进来
      （只交 `cases/` 下的终态产物即可）

## 红 / 黄 / 灰 对照表

| 颜色 | 含义 | 例子 | 要不要改 |
|---|---|---|---|
| 🔴 红色 | 严重问题，阻止提交 | `artifact_hash_mismatch`（产物被改过）、`chain_break`（账本断链）、`hash_mismatch`、`agent_direct_write`、`artifact_missing` | **必须修** |
| 🟡 黄色 | 建议修正 / 需复核 | `case_state_path_should_be_teaching_root`（路径写法）、`artifact_not_in_ledger`（产物没账本背书） | 尽量改；改不掉如实说明 |
| ⚪ 灰色 | 无害软提示 | `output_superseded`（重跑留下的旧记录）、`repo_root_unresolved`、`input_*` | 忽略 |

## 关于"会不会被判作弊"——放心吃定心丸

- **失败本身不扣分。** 如实记录的 `FAILED_* / BLOCKED_* / *_NOT_RUN / INSUFFICIENT_EVIDENCE`，
  只要产物和状态自洽，都是有效证据。
- 真正会出问题的只有这几种：**无证据的 PASS**、**伪造产物**、**切平台**（红线 3）、
  **抄袭**（和别人交了相同内容、跨提交撞哈希）、**手工搬入的无账本产物**。
- 所以——**老实跑、如实记**，永远比"修得好看"安全。跑不通就如实写 BLOCKED，留好证据。
